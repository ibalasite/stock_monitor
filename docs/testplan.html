§1 測試目標

01

驗證 EDD 所有業務規則：Schema constraint、訊號優先級、冷卻邏輯、冪等鍵

02

驗證 通知/冷卻/補償鏈完整性：LINE 成功/失敗路徑、pending ledger 補償

03

驗證 14:00 估值排程在交易日執行、非交易日 skip、失敗不覆蓋快照

04

驗證 交易時段邊界：08:45 開盤檢查、09:00–13:30 監控、13:31 後停止

05

驗證 開盤摘要：同日第一個可交易分鐘推送、同日不重複、重啟仍受 DB 保護

06

TDD 達成 lines/branches/functions/statements 全部 100%

07

驗證 LINE 模板渲染：所有出站訊息由 template_key + context 產生

08

驗證 Code Review 改善項目（CR-SEC-01~03、CR-ARCH-01~03、CR-ADP-01~02）

09

驗證 Freshness-First 雙行情來源：Yahoo 容錯、tick_at 比較、HTTP 安全上限

§2 測試範圍

✅ 在範圍（In Scope）

  • Schema/migration（TP-DB）
  • Domain policy：訊號優先級、冷卻、冪等（TP-POL）
  • 盤中主流程：抓取 → 評估 → 彙總 → 推送 → 落盤（TP-INT）
  • LINE 成功/失敗路徑（TP-INT-004/-005）
  • 補償機制 Pending Ledger（TP-INT-006/-008）
  • 交易日判定與時段邊界（TP-TRD）
  • 每日估值排程（TP-VAL）
  • TimeBucket 服務（TP-BKT）
  • 通知準確度 KPI（TP-KPI）
  • 開盤摘要（TP-INT-012 / TP-UAT-013）
  • LINE 模板渲染（TP-TPL）
  • 安全/架構/程式碼品質（TP-SEC / TP-ARCH / TP-CODE）
  • 雙行情來源 Adapter（TP-ADP）

❌ 不在範圍（Out of Scope）

  • 自動下單功能
  • 分散式高可用部署
  • 非台灣市場（美股、加密）
  • 生產 LINE token 實際推送（CI 環境使用 mock）
  • UI/UX 介面測試
  • 多租戶或多資料庫部署

TP-SEC-003

TP-ARCH-001

app.py 不含估值邏輯；無 scenario_case 生產路徑

TP-ARCH-002

render_line_template_message 全專案唯一定義

TP-ARCH-003

MinuteCycleConfig dataclass 在 runtime_service.py

app.py 不定義 _run_daemon_loop / _build_runtime；只保留 CLI 入口與指令路由

TP-ARCH-006

merge_minute_message 改私有（_merge_minute_message）；無公開孤兒 symbol

TP-ADP-001

Yahoo HTTP 失敗 → WARN + 空 dict，不 raise

TP-ADP-004
