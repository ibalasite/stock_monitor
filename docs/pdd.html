產品規格文件（PDD）

版本 v1.0  ·  2026-04-14  ·  狀態：Draft（可進入 review）
定義「台股價格監控 + LINE 通知系統」之產品需求，讓工程與企劃可共同對齊。

本文件對齊 EDD、TEST_PLAN、USER_STORY_ACCEPTANCE_CRITERIA 三份文件。 如有衝突，以「可執行規格（feature + tests）」為最高優先。

§2 背景與問題

手動盯盤耗時

使用者需全程看盤，容易錯過理想買點，且長時間盯盤難以維持。

決策延遲成本高

市場波動快，若無自動化通知，進場時機往往已過。

通知疲勞問題

同訊號重複通知過多，反而使用者忽略真正重要的訊號。

§3 產品目標

業務目標

  • 降低手動盯盤時間
  • 在價格進入可行動區間時即時提醒

成功指標（初版）

指標目標值
通知延遲觸發後 60 秒內送達（P95）
通知準確率> 99%（非資料源中斷情況）
重複通知控制相同訊號 5 分鐘內不重複發送

§4 目標使用者

  • 個人投資者（優先）— 主線使用情境
  • 小型投資團隊（後續擴展）— 群組通知模式

§5 產品範圍

In Scope — V0
  • 監控清單維護
  • 每分鐘行情輪詢
  • 合理價/便宜價訊號判斷
  • 5 分鐘通知冷卻
  • LINE Bot 群組推播
  • SQLite 本機儲存
  • 啟動 fail-fast 驗證
In Scope — V1
  • 14:00 每日估值計算
  • 三種估值方法
  • 方法 enable/disable
  • 多方法同分鐘彙總通知
  • 開盤監控設定摘要通知
  • 雙行情來源（TWSE + Yahoo）
  • LINE 訊息模板化（FR-14）
Out of Scope
  • 自動下單
  • 多市場（美股/加密）
  • 分散式高可用叢集

§6 使用流程（User Flow）

  1. 使用者設定監控股票與手動合理價/便宜價。
  2. 交易日開盤起始先推播 1 封「當日監控設定摘要」（股票、方法、合理價、便宜價）。
  3. 系統在交易時段（09:00–13:30）每 1 分鐘抓委賣一價格。
  4. 若現價低於合理價或便宜價，立即推播 LINE 到指定群組。
  5. 5 分鐘內同股票同狀態不再推播，且 message.update_time 不更新。
  6. 每交易日 14:00 執行估值計算並入庫。
  7. 隔日盤中依各估值方法門檻持續監控並通知。

§7 功能需求（FR）

訊號判斷規則

status觸發條件優先序
status=1market_price ≤ fair_price（低於合理價）次要
status=2market_price ≤ cheap_price（低於便宜價）最高
若同時符合 status=1 與 status=2 → 只發送 status=2（便宜價優先）。 達便宜價必然達合理價，status=2 才是當下最重要訊號。

雙行情來源（FR-15）

行情 price 代表委賣一（最佳委賣價）— 當下可立即買入的最低價格。 成交價（z）在兩筆成交之間短暫閃爍為 '-'，委賣一維持連續更新，更能反映現況。

來源端點欄位備註
主 TWSE mis.twse.com.tw getStockInfo.jsp a 欄第一值 委賣一；tlong 毫秒時間戳；含 _price_cache 補全機制
副 Yahoo tw.stock.yahoo.com/quote/{stock_no} HTML 委賣一 HTML scraping；盤後 fallback regularMarketPrice；失敗不中斷主流程

Freshness-First 取捨規則

  • Rule 1TWSE cache 有值且 Yahoo regularMarketTime ≤ TWSE tick_at → 採 TWSE
  • Rule 2Yahoo regularMarketTime 嚴格 > TWSE tick_at → 採 Yahoo
  • Rule 3TWSE cache 空（冷啟動第一個輪詢）→ 直接採 Yahoo
  • Rule 4兩者均無法取得有效價格 → 標記 STALE_QUOTE，跳過通知

通知冷卻（FR-05）

項目規格
冷卻鍵（維度）stock_no + stock_status
冷卻時間300 秒（5 分鐘）
冷卻中行為不發送、不更新 message.update_time
同分鐘冪等鍵stock_no | minute_bucket（不含 stock_status）

每日估值結算（FR-07 / FR-08）

艾蜜莉 複合法

emily_composite_v1

股利法、歷年股價法、本益比法、淨值比法取平均，再乘安全邊際係數（0.9）。

股海老牛 殖利率法

oldbull_dividend_yield_v1

平均股利 ÷ 5% = 合理價；平均股利 ÷ 6% = 便宜價。

雷司紀 四法融合

raysky_blended_margin_v1

PE、股利、PB、NCAV 取中位數或加權融合，套安全邊際係數（0.9）。

開盤摘要通知（FR-13）

  • 觸發時機：每交易日第一個可交易分鐘（通常 09:00）
  • 內容：逐股票逐方法 fair_price / cheap_price
  • 股票格式：中文名(代號)（例：台積電(2330)）
  • 無快照方法顯示 N/A
  • 同一交易日僅允許發送一次，服務重啟不得重複推送
  • 必須使用模板渲染（FR-14），不得硬編碼

補償機制（FR-06）

LINE 發送成功 + DB 落盤失敗時，寫入 pending_delivery_ledger， 補償完成前視同已通知，避免重複推送同分鐘訊號。

所有功能需求清單

  • FR-01監控清單管理：新增/停用/刪除，含 stock_no / manual_fair_price / manual_cheap_price / enabled。
  • FR-02盤中每 60 秒輪詢，交易時段限定；失敗重試 MAX_RETRY_COUNT=3；stale/衝突時跳過且不補發。
  • FR-03訊號判斷：status=1（≤fair）/ status=2（≤cheap）；同時命中僅發 2。
  • FR-05通知冷卻：stock_no + stock_status 維度，5 分鐘內不發送也不更新時間。
  • FR-06message 表唯一鍵 UNIQUE(stock_no, minute_bucket)；LINE 失敗僅寫 log；LINE 成功 DB 失敗寫補償佇列。
  • FR-07每交易日 14:00 估值結算，失敗不覆蓋舊資料；寫入 fair/cheap/方法/版本/交易日。
  • FR-08多方法估值：全域 enable/disable；同方法名稱同時僅一個啟用版本；多方法命中附方法清單。
  • FR-09啟動前驗證 LINE_CHANNEL_ACCESS_TOKEN / LINE_TO_GROUP_ID；無效即 fail-fast；不得輸出 token 明文。
  • FR-10重啟後先載入最近通知狀態再輪詢；補償中訊號可續跑，不重複推送同分鐘。
  • FR-11三估值方法定義：emily_composite_v1 / oldbull_dividend_yield_v1 / raysky_blended_margin_v1。
  • FR-12估值結果三態：SUCCESS / SKIP_INSUFFICIENT_DATA / SKIP_PROVIDER_ERROR；單方法失敗不阻斷整體任務。
  • FR-13開盤監控設定摘要通知：每交易日首個可交易分鐘發送一次，內含各股各方法 fair/cheap。
  • FR-14LINE 訊息全模板化：所有對外訊息皆由 template_key + context 渲染，不得硬編碼文案。
  • FR-15雙行情來源：TWSE MIS 主（委賣一 a 欄）+ Yahoo Finance HTML 副（委賣一 scraping）。
  • FR-16MarketDataPort 介面可注入替換；CompositeMarketDataProvider 實作 Freshness-First 聚合。

類別規格

§10 資料結構（摘要）

watchlist

監控股票清單；manual_cheap_price ≤ manual_fair_price

valuation_methods

估值方法版本表；同名稱同時僅一個 enabled=1

valuation_snapshots

每日估值結果；唯一鍵含 method_version

message

通知紀錄；UNIQUE(stock_no, minute_bucket)；冷卻查詢依賴

pending_delivery_ledger

補償佇列；status: PENDING / RECONCILED / FAILED

system_logs

系統日誌；level: INFO / WARN / ERROR

§11 Clean Architecture 建議設計

層次元件
Domain Entities: Stock, PriceSignal, ValuationResult, MessageRecord

Ports: MarketDataPort, LineMessagingPort, MessageRepositoryPort, ValuationRepositoryPort

§12 UAT 驗收標準（15 條）

  • UAT-01設定 2330 fair=1500, cheap=1000；當價格 ≤ 1500 或 ≤ 1000 時，LINE 在 60 秒內收到群組通知。
  • UAT-02相同 2330 + stock_status 5 分鐘內不重複推送，message.update_time 不更新。
  • UAT-03message 表可查到 stock_no / message / stock_status / update_time。
  • UAT-04非交易時段（含週末與假日）不進行盤中輪詢。
  • UAT-05每交易日 14:00 估值任務執行一次；失敗不覆蓋舊估值。
  • UAT-06同一分鐘多股票/多方法同時觸發，系統只發 1 封彙總訊息，內含所有命中方法資訊。
  • UAT-07同一股票同分鐘同時符合 status=1 與 status=2，僅以 status=2 呈現與通知。
  • UAT-08「LINE 成功 + DB 落盤失敗」時，補償流程可回補；補償期間不重複發送同分鐘訊息。
  • UAT-09缺少或誤設 LINE_CHANNEL_ACCESS_TOKEN / LINE_TO_GROUP_ID，系統啟動 fail-fast 並輸出明確錯誤原因。
  • UAT-10服務重啟後，已送出的同分鐘訊號不得再次發送；補償中訊號可續跑。
  • UAT-11行情 stale 或來源衝突，該分鐘不得通知，需有 STALE_QUOTE 或 DATA_CONFLICT 的 WARN 日誌。
  • UAT-12每交易日 14:00 對三方法逐股估值；資料不足方法 skip + log，不覆蓋舊快照。
  • UAT-13每交易日開盤第一個分鐘先發「監控設定摘要」；同一交易日不重複；由模板渲染。
  • UAT-14所有 LINE 訊息皆透過 template_key + context 渲染；程式碼中不得直接硬編碼最終文案。
  • UAT-15雙行情來源 Freshness-First：Yahoo 失敗不中斷主流程；兩者均無時 STALE_QUOTE。

§13 風險與因應

風險因應措施
資料源中斷加重試（MAX_RETRY=3）、Freshness-First fallback、WARN 日誌
通知與落盤不一致補償佇列 + 回補重試，避免重複告警與狀態遺失
行事曆資料延遲以「無新大盤資料」作次級判斷避免誤開盤
設定錯誤啟動前驗證 + fail-fast，明確錯誤訊息

§14 里程碑

  • M1（2–3 天）V0 可用：手動 fair/cheap 監控 + LINE 通知 + cooldown。
  • M2（2–4 天）V1 估值快照入庫 + 多方法 fair/cheap 監控。
  • M3（1–2 天）補齊監控報表與操作文件。
